# THINK - Deep Analysis and Systematic Problem Solving

## Overview
The THINK skill enables deep, structured analysis of complex problems. It guides you through systematic decomposition, root cause identification, and solution evaluation to arrive at well-reasoned conclusions.

## Workflow

### 1. Problem Definition
- Clearly state the problem in your own words
- Identify what success looks like (measurable outcomes)
- Determine constraints and non-negotiables
- Separate symptoms from root causes
- Define the scope boundaries (what is in/out of scope)

### 2. Context Gathering
- Collect all relevant information and data
- Identify stakeholders and their perspectives
- Map dependencies and relationships between components
- Note any assumptions being made and validate them
- Research prior art or similar solutions if applicable

### 3. Decomposition
- Break the problem into smaller, manageable sub-problems
- Identify the critical path and dependencies between sub-problems
- Classify each sub-problem: technical, design, business, or operational
- Prioritize by impact and urgency using a matrix (high/low impact x high/low effort)

### 4. Analysis
- For each sub-problem, identify multiple possible approaches
- Evaluate trade-offs for each approach: complexity, maintainability, scalability, cost
- Consider edge cases and failure modes for each option
- Use first principles thinking: strip away assumptions and reason from fundamental truths
- Apply mental models where relevant (inversion, second-order thinking, occam's razor)

### 5. Synthesis
- Combine the best elements from analyzed approaches
- Construct a coherent solution that addresses all sub-problems
- Validate the solution against the original problem definition
- Identify any gaps or unresolved concerns
- Document the reasoning behind key decisions

### 6. Validation
- Stress-test the solution against edge cases
- Consider: What could go wrong? How would we detect it? How would we recover?
- Seek contradictions or logical gaps in the reasoning
- If possible, create a small proof-of-concept or mental simulation
- Review assumptions made during analysis and verify they still hold

## Best Practices
- Write out your reasoning explicitly; do not skip logical steps
- Question assumptions at every stage; mark unverified assumptions clearly
- Use structured formats (lists, tables, diagrams) to organize complex information
- When stuck, try explaining the problem to a rubber duck or writing it differently
- Balance thoroughness with pragmatism; not every problem needs exhaustive analysis
- Document key decision points so the reasoning can be reviewed later

## Output Format
When presenting analysis results, structure as:
1. Problem statement (1-2 sentences)
2. Key findings from context gathering
3. Decomposition breakdown (bullet list of sub-problems)
4. Analysis summary (pros/cons table for major decisions)
5. Recommended solution with reasoning
6. Open questions and risks
